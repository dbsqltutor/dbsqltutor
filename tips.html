SQL Tip #1: Why SELECT * Hurts Performance More Than You Think

Ever wonder why a query feels slow even when the table isn’t that big? A common culprit is SELECT *. It looks harmless, but it quietly creates several performance problems.

What’s really happening when you use SELECT *:

  • You return every column from every table in the query — even ones you don’t need. In joins, this can dramatically increase the total amount of data returned.
  • More data means more I/O, more memory use, and more work for the database engine.
  • Wide datatypes (JSON, XML, VARCHAR(MAX), BLOB) make the slowdown even worse.
  • The optimizer may be unable to use covering indexes, forcing extra lookups and slowing execution.

A simple rule of thumb:

Only select the columns you actually need.
It keeps queries faster, reduces memory use, and makes your intent clearer to anyone reading your code.

Quick example:

Instead of:
SELECT * FROM Orders

Try:
SELECT OrderID, OrderDate, TotalAmount FROM Orders

If you want to build better SQL habits, start by replacing SELECT * with the specific fields you need. It’s a small change that pays off immediately.

Database Tip #1: Why Choosing the Right Data Types Matters

Data types matter far more than most people realize. Oversized or inconsistent definitions can lead to wasted storage, slower reads, and weaker data integrity.

What goes wrong when using the wrong data type:

  • Poor integrity: Storing dates as VARCHAR allows inconsistent formats like 1/10/2025, 2025/10/1, or January 10, 2025.
  • Invalid values: A VARCHAR column meant for ages could accept values like 'twenty', '20', or '999'.
  • Slower queries: Comparing text to numbers forces type conversion and may prevent index use.
  • Inefficient sorting/filtering: Sorting strings like '10', '2', '100' produces incorrect order compared to numeric sorting.
  • Wasted storage: Using CHAR(50) for a 2‑digit code inflates row size and reduces page density; VARCHAR is more appropriate.
  • Weaker validation: A VARCHAR column meant for prices can’t prevent entries like 'free' or 'ten dollars'.

When using the wrong data length:

For example, storing a person’s age as NUMBER(2) is sufficient. Defining it as NUMBER(5) instead can cause unnecessary overhead:

  • Extra storage for a wider numeric definition.
  • Larger index entries, which can slow index scans and lookups.
  • Reduced block/page density because fewer rows fit into each data block.
  • More work for the engine to handle a higher‑precision numeric type than needed.
  • Increased memory and cache usage when moving and comparing the wider value.

Two simple rules of thumb:

  1. Match the data type to the nature of the data.
    CHAR/VARCHAR for text, NUMBER/INT/DECIMAL for numeric values, DATE/DATETIME for dates and timestamps.
  2. Specify a maximum length that makes sense.
    For example, using NUMBER(10) for a person’s age is excessive — age will never exceed three digits.
